# long-ppt-core

`long-ppt-core` 是 `Long-ppt skill` 体系里的总控层。

如果只看一个子 skill 来理解整套 Long-ppt，应该先看它。

它负责的不是单一技术动作，而是把“现有材料 -> 内容理解 -> 结构提炼 -> 路线判断 -> PPT 交付”这条链路串起来。

## 它解决的核心问题

很多 PPT 任务真正难的不是画，而是判断：

- 现有材料到底在讲什么
- 应该保留什么，重写什么
- 原来的大纲是否成立
- 哪些页该先重设计
- 哪些页该保留截图
- 哪些页该做原生可编辑重建

`long-ppt-core` 就是负责做这些判断的。

## 最适合它的输入材料

它尤其适合处理这些输入：

- 现有 PPT
- Word 方案文档
- 项目材料
- 原始汇报稿
- 零散笔记
- 直播讲稿 / 演讲提纲

也就是说，如果用户说的是：

- “我给你一套 PPT 和 Word，帮我整理”
- “帮我总结现有材料，重写大纲”
- “帮我从原稿改成新的演讲版 PPT”

最核心对应的就是 `long-ppt-core`。

## 它负责哪些事情

- 基于现有材料理解内容
- 提炼页面结构和信息层级
- 重写目录和页面大纲
- 判断哪些内容适合压缩、拆分或重组
- 判断每一页该走哪条生产路线
- 把后续的 image / hybrid / native / vba 路线串起来

## 它和其他子 skill 的关系

- `long-ppt-core`
  - 负责理解材料、重写大纲、判断路线
- `long-ppt-image`
  - 负责先出视觉图、底板、替代示意图
- `long-ppt-hybrid`
  - 负责截图型页面的半可编辑重建
- `long-ppt-native`
  - 负责架构图、框架图、附件页等结构化原生重建
- `long-ppt-vba`
  - 负责宏驱动环境下的原生重建

## 结构提炼是什么意思

在 Long-ppt 里，很多页面不会直接开始画，而是先做一层结构整理：

- 这页真正的标题是什么
- 哪些是主信息
- 哪些是解释信息
- 哪些区域是视觉重点
- 哪些区域适合改成原生对象

只有这一步对了，后面的生成和重建才不会跑偏。

## 默认判断顺序

一般建议按这个顺序判断：

1. 页面视觉本身不够好，先走 `long-ppt-image`
2. 页面像截图、大屏、仪表盘，优先走 `long-ppt-hybrid`
3. 页面像架构图、框架图、附件页，优先走 `long-ppt-native`
4. 当前流程依赖宏自动化，再考虑 `long-ppt-vba`

## 一句话定位

`long-ppt-core` 是面向 PPT / Word / 方案材料的内容理解、结构提炼、大纲重写与整套路线编排能力。
