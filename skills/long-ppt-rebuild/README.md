# long-ppt-rebuild

`long-ppt-rebuild` 是 `Long-ppt skill` 里专门处理“现有视觉页 -> 可编辑 PPT”重建任务的一条路线。

它承接的是这类输入：

- 一张现有的 PPT 截图
- 一页 PNG / JPG / WebP 视觉稿
- 图片版 PPTX
- 需要从现有视觉页还原成可编辑 PPT 的页面

## 它和别的子 skill 的关系

这条路线和其他几支的区别很明确：

- `long-ppt-image`
  - 是先生成新图或底板
- `long-ppt-hybrid`
  - 是保留复杂视觉，再把关键内容做成可编辑
- `long-ppt-native`
  - 是结构化页面的原生对象重建
- `long-ppt-rebuild`
  - 是以“现有视觉页”为起点，做语义化、可编辑化的整页重建

可以把它理解成：

`long-ppt-rebuild = 面向现有图片页 / 截图页 / 整页视觉稿的可编辑 PPT 还原流程`

## 最适合它的场景

- 用户给的是一页现成图片，不是结构提纲
- 用户明确说“把这页做成可编辑 PPT”
- 页面既有文字，也有卡片、箭头、标签、视觉元素
- 原始文件已经变成图片页，但仍想恢复可编辑能力

## 它的核心方法

不是盲目描图，而是：

1. 先做视觉清单
2. 再判断哪些元素该原生化
3. 哪些元素该作为独立图片资产
4. 最后再重建成 PPTX

也就是说，它强调的是：

- 语义重建
- 实用可编辑
- 交付可验证

## 关键原则

- 文本、卡片、标签、普通线条、简单箭头，应优先做成原生对象
- 图标、装饰图、复杂箭头带、复杂视觉块，应作为独立图片资产处理
- 最终交付必须能回到 PPT 里继续改，而不是只得到一张大图

## 和 `ppt2` 的关系

这条路线就是把你现有的 `ppt2` 思路，整理进 `Long-ppt` 体系后的结果。

保留的核心思想包括：

- 从现有视觉页出发
- 做语义级重建，不是简单截图覆盖
- 强调可编辑对象与独立视觉资产的分类
- 交付前必须做渲染验证

## 一句话定位

`long-ppt-rebuild` 是面向现有图片页、截图页、整页视觉稿的“语义化可编辑 PPT 重建能力”。
