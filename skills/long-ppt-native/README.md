# long-ppt-native

`long-ppt-native` 是 `Long-ppt skill` 里专门负责“原生可编辑重建”的一条路线。

它的核心不是 VBA，也不是把图片贴回 PPT，而是直接使用 `python-pptx` 输出 PowerPoint 原生对象：

- 文本框
- 矩形 / 圆角矩形
- 连线 / 箭头
- 标签
- 分区容器

最终得到的是普通 `pptx`，不是图片版 PPT，也不是必须依赖宏的 `pptm`。

## 这条路线最适合什么

最适合下面这类页面：

- 架构图
- 附件页
- 结构化方案页
- 框架图
- 能力地图
- 分层平台图
- 规则明确、布局稳定的结构化页面

如果页面本质上是“大块结构 + 模块卡片 + 连线关系 + 说明文字”，通常都应该优先考虑 `long-ppt-native`。

## 它和别的路线怎么区分

### 和 `long-ppt-hybrid` 的区别

`hybrid` 更适合：

- 截图页
- 大屏页
- 监控页
- 仪表盘页

这类页面通常图片信息密度高，不值得全部原生重画。

`native` 更适合：

- 结构图
- 框架图
- 模块图
- 可继续维护的架构页

### 和 `long-ppt-vba` 的区别

`vba` 更适合宏驱动的办公环境。

`native` 更适合：

- 直接产出标准 `pptx`
- 不依赖 Office 宏环境
- 用脚本稳定批量重建

## 典型结果

这条路线的典型输出会具备这些特征：

- 是普通 `pptx`
- 可以直接打开编辑
- 页面由 PowerPoint 原生对象组成
- 没有把整页当作一张底图贴进去
- 后续可以继续改文字、改模块、改配色、改层级

## 真正的关键不在“描图”，而在“结构提炼”

`long-ppt-native` 最重要的能力，不是把图片描成形状，而是先做一层结构提炼：

- 这一页有几层
- 每层有哪些模块
- 左右关系是什么
- 连线关系是什么
- 颜色系统是什么
- 哪些元素应该抽象成可复用组件

也就是说，这条路线本质上是：

`现有材料 / 原图 -> 结构建模 -> 原生对象重建`

而不是：

`图片 -> 盲目自动矢量化`

## 推荐工作流

通常建议这样使用：

1. 先由 `long-ppt-core` 判断这页是否适合原生重建
2. 如果不能直接用真实项目图，可先由 `long-ppt-image` 生成不泄密的示意图
3. 再进入 `long-ppt-native`，按结构输出原生可编辑 PPT

## 真实案例

### 案例 1：电力可观测性架构图可编辑重建

这是一个典型的 `long-ppt-native` 案例：

- 输出 PPT：
  - `D:\Projects\southwest_pipeline_colasoft\output\电力可观测性架构图_可编辑重建版_v1.pptx`
- 预览图目录：
  - `D:\Projects\southwest_pipeline_colasoft\output\preview_power_observability_editable_v1`

这个案例最能体现这条路线的价值：

- 目标不是重新设计一张新图
- 而是基于既有结构做高保真可编辑重建
- 最终结果能继续改模块、改说明、改结构

### 案例 2：替代真实项目图的结构化重建

当真实项目图不能直接展示时，也可以先生成一张“结构类似但不泄密”的示意图，再用 `python-pptx` 重建成可编辑页。

## 什么时候优先选它

下面这些情况，优先考虑 `long-ppt-native`：

- 用户要的是正常 `pptx`
- 后续还要大量改字和改模块
- 页面属于结构化图示
- 页面不是截图型大屏
- 页面不想依赖 VBA

## 一句话定位

`long-ppt-native` 是面向架构图、框架图、附件页和能力地图的 `python-pptx` 原生可编辑重建能力。
