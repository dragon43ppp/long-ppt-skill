# 案例：AI 架构图到可编辑 PPT

这个案例用于说明 `Long-ppt` 的一个典型工作流：

1. 先用图片模型生成“示意性架构图”
2. 再把它转成可编辑 PPT

## 目标

适用于下面这类需求：

- 不能直接展示真实项目图
- 需要一张“风格接近、结构类似、但不泄密”的替代示意图
- 后续还要继续修改标题、模块、标签和连接关系

## 正确理解这条路线

这条路线的重点不是“自动描边”。

更稳妥的方式通常是：

1. 先从原图或需求中提炼结构
2. 抽象出层级、模块、连接关系、配色和信息密度
3. 用 `gpt-image-2` 生成一张新的示意图作为视觉参考
4. 再用 `python-pptx` 或 VBA 按结构重建成 PowerPoint 原生对象

## 为什么不是直接反向拆图

因为企业架构图最有价值的部分通常不是像素，而是：

- 层级关系
- 模块命名
- 业务分组
- 连线逻辑
- 可继续编辑的结构

如果直接把图片“描成形状”，往往会出现：

- 形状数量很多，但结构不可维护
- 文字层不完整
- 连线关系不清晰
- 后续改版成本很高

## 推荐实现方式

### 路线 A：结构化重建

最推荐。

做法：

1. 先提炼出标题、层级、模块、说明框、连接线
2. 封装通用组件，例如：
   - 标题
   - 区块卡片
   - 标签
   - 说明框
   - 连线
3. 使用 `python-pptx` 输出原生对象

优点：

- 输出就是普通 `pptx`
- 无图片叠加
- 可继续编辑
- 适合公开案例与可复用脚本

### 路线 B：Hybrid 重建

适合大屏、监控截图、复杂界面页。

做法：

- 视觉复杂区域保留为图片
- 标题、长文本、标签、卡片说明改为可编辑对象

优点：

- 视觉保真度高
- 迭代快

### 路线 C：VBA 重建

适合已经依赖 PowerPoint 宏的办公流。

优点：

- 可以直接在 PowerPoint 端执行
- 适合宏批处理与内部办公环境

## 当前工程上的现实结论

如果 `gpt-image-2` 接口链路可用，这条流程是完整可走通的。

如果图片代理层暂时不稳定，`Long-ppt` 仍然可以先完成：

- 结构提炼
- 原生 PPT 重建
- 可编辑对象交付

也就是说，图片生成只是视觉入口，不是整套能力成立的前提。

## 一个实用建议

如果你的目标是“公开给别人用”，需要把图片调用层设计成可替换的执行层，而不是写死某一个代理实现。

推荐做法：

- Skill 文档负责说明页面判断与工作流
- 图片执行层支持：
  - 官方 OpenAI
  - OpenAI 兼容转发
  - Azure OpenAI
  - 项目自定义代理

## 对应的 skill 映射

- 页面路线判断：`long-ppt-core`
- 视觉示意图生成：`long-ppt-image`
- 原生对象重建：`long-ppt-native`
- 宏路线补充：`long-ppt-vba`
